## Confirmed business rules (REVISED 2026-07-11: aggregator model)

- **Signup**: public form, but tenant activates only after Daniel approves it in the
  platform portal (status: pending → approved → active → suspended).
- **AGGREGATOR MODEL** (supersedes per-ISP paybills — most small ISPs lack BRS
  registration): ALL customer money flows through **Danamo Tech Ltd's** paybill and
  Daraja app. ISPs need zero Safaricom paperwork. Tenant attribution via the
  `AccountReference` (tenant slug) and the operator FK on every transaction.
- **Wallets (apps/billing)**: every successful sale credits the tenant's ledger with
  gross amount; the platform commission (default 3%, per-tenant rate) is **withheld at
  source** as a debit line. Base fee auto-deducts from the wallet on the 1st
  (idempotent per month). PPPoE per-user fee joins as a wallet deduction in Phase 3.
- **Payouts v1**: ISP requests a withdrawal (funds held immediately, min KSh 100) →
  Daniel's payout queue → he sends the M-Pesa manually and records the transaction
  code (or rejects, refunding the wallet). B2C automation later.
- Regulatory note: aggregating third-party payments touches CBK/PSP rules — get
  advice as volumes grow.
- **PPPoE v1 is full-control**: the system creates PPPoE secrets AND speed profiles on
  the router per plan; suspend/restore is automatic on invoice status. PPPoE C2B
  account numbers (WIF-XXXXX) resolve on Danamo's paybill.

## Phase 1 — Tenancy core

- `TenantMiddleware`: resolve Operator from Host subdomain; `request.operator` everywhere.
- Kill `get_default_operator()` call sites: every queryset/creation scoped to the
  request's operator. Staff users belong to exactly one operator (platform staff = none).
- ISP signup API + approval flow; onboarding wizard UI: business details → paybill +
  Daraja creds (validated with a live token request) → first plans from templates →
  router onboarding script → done.
- Per-tenant portal context: captive portal reads tenant from router record; each
  router belongs to an operator, so payments/plans resolve from the router param.
- Dev-mode tenancy: subdomains via `<slug>.localhost` so everything is testable locally.

## Phase 2 — Router self-onboarding + resilience

- "Add Router" generates a **one-paste RouterOS script** per router containing:
  API user creation, WireGuard peer config (keys generated server-side; hub activates
  at deployment — LAN direct mode until then), hotspot + walled garden + login page
  fetch, and a phone-home call to
  `POST /api/v1/routers/<id>/register/?token=...` that flips the router to online.
- **Re-sync on reconnect**: when health check sees offline→online, run
  `sync_router_sessions`: DB sessions that should be ACTIVE but missing on the router
  get re-provisioned; orphan hotspot users on the router get removed. Also run nightly.
- Router health history (uptime %, last 24h) for both ISP and platform dashboards.

## Phase 3 — PPPoE + C2B (biggest phase)

- Models: PPPoE subscriber profile (User + plan + router + unique
  `account_number` e.g. WIF-XXXXX), monthly Invoice (period, due date, status),
  auto-generated by beat on each subscriber's billing day.
- **C2B**: register validation + confirmation URLs per tenant paybill (Daraja
  registerurl API); confirmation matches `BillRefNumber` → account number → invoice →
  mark paid → restore/keep service. Store raw payloads; idempotent on TransID.
- Provisioning: `activate_pppoe` / `suspend_pppoe` in the MikroTik adapter
  (`/interface/pppoe`… secrets + `/ppp/profile` rate limits per plan).
- Auto-suspend on overdue (grace days configurable per tenant), auto-restore on payment.
- ISP UI: PPPoE clients screen (create client → system assigns account number →
  printable/SMS-able payment instructions), invoices screen.

## Phase 4 — Platform portal + platform billing

- New frontend `platform-ui/` (same design system): tenants list + approval queue,
  tenant detail (routers online, active users, revenue, MRR), **ISP→platform tickets**,
  platform invoices, global metrics.
- Metering beat job (1st of month): for each tenant compute base fee + 3% hotspot
  commission + PPPoE user count × rate → PlatformInvoice; STK/C2B against Daniel's
  paybill; dunning → suspension.
- Platform roles: Daniel = superuser; support staff read-only.

## Phase 5 — Load & resilience verification (exit criteria for deployment)

- k6/locust suite: (a) 500 concurrent portal sessions browsing plans; (b) 100
  STK initiations/min with callback bursts; (c) callback replay storms (idempotency
  under concurrency, not just sequential); (d) 20 tenants × health checks × expiry
  sweeps concurrently.
- Router kill test: power-cycle the RB951 mid-session → verify auto re-sync.
- Targets documented in `docs/CAPACITY.md` with measured numbers.

## Explicitly deferred to deployment phase

(apply early — Safaricom review takes time), backups off-box, Sentry.
